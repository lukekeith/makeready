# Media Library at Scale — Implementation Plan (2026-06-10)

| | |
|---|---|
| **Requirement** | Orgs will realistically hold **tens of thousands** of media items, possibly **hundreds of thousands to millions**. The library must stay native-Photos smooth at that scale. |
| **Context** | Structural Phase 4 (audit plan, `docs/audit/iphone-2026-06-10-plan.md`) shipped the rendering layer: paginated loading, diffable grid, prefetching, in-flight coalescing, sized decode, delegate sizing, restore-closure handshake. That layer survives this plan intact. What does **not** scale is the data layer behind it. |
| **Prime directive** | Same as the audit plan for M0–M1: zero functionality loss, zero design change. M2 introduces a new data layer (invisible). M3 is the visible, design-led redesign and is explicitly out of "structural" scope. |

---

## Why the current architecture caps out

Current flow: `MediaActions.fetchLibrary(page:)` → Prisma `skip/take` offset query → items accumulate in `AppState.mediaLibrary` (an in-memory `EntityStore`) → **every** loaded item is re-encoded into the `PersistedState` JSON snapshot on each debounced `persist()` → `MainLibrary.filteredMedia` sorts the full loaded array per body evaluation → diffable snapshot carries every loaded id.

| Component | Breaks at | Failure mode |
|---|---|---|
| `PersistedState` JSON snapshot | ~5–10k items | Multi-second full-library JSON encode on every debounced save; giant `app_state.json`. **Newly reachable**: Phase 4.1 removed the 100-item cap, so deep scrolling now grows the snapshot without bound. |
| In-memory `EntityStore` | ~50–100k | ~1KB/model × every item ever scrolled past; nothing evicts. |
| `filteredMedia` sort | ~50k | O(n log n) over the full loaded set per render evaluation, on the main thread. |
| Offset pagination (`skip/take`) | ~100k | Page 1,000 makes Postgres scan and discard 99,900 rows per request; latency grows linearly with depth. |
| Diffable id snapshots | ~100k | Six-figure id-array diffs on the main thread per append. |
| Disk image cache | unbounded | No eviction; a million-item org slowly fills the device. |

**The Photos model (target):** Photos binds its grid to a local SQLite database and *faults* — the UI knows only a count and materializes items near the viewport, evicting behind the scroll. Sort/filter run in the database, never over an in-memory array. Thumbnails come in size tiers with a budgeted cache. The cloud-backed equivalent for MakeReady is a local metadata store synced incrementally by cursor, with `AppState` holding only the visible window.

## Phase M0 — Immediate guardrails (client-only, behavior-preserving)

> Closes the hazard Phase 4.1 introduced. Each step is invisible and independently committable. Effort: ~1 session. Can be slotted as a decimal phase (e.g. 4.8) without disturbing the structural roadmap.

| # | Step | Detail | Invisibility guarantee |
|---|---|---|---|
| M0.1 ✅ shipped 2026-06-11 | **Cap the persisted media snapshot at one page (100 items)** | `PersistedState.init(from:)` takes `Array(appState.orderedMedia.prefix(MediaActions.libraryPageSize))` instead of `mediaLibrary.all`. | Pre-Phase-4 builds never persisted more than 100 items (the old cap); this restores that exact launch behavior. Cold launch shows the first page from disk, network refresh follows — unchanged. |
| M0.2 ↪ **folded into M2** (2026-06-11) | **Sliding-window eviction in `AppState.mediaLibrary`** | When loaded count exceeds a window (e.g. 1,000 = 10 pages), evict items more than N pages behind the current scroll anchor. **Why deferred:** `MediaLibraryGrid` is a UICollectionView driven directly by the resident items array — there are no placeholder slots, so evicting items collapses rows and visibly jumps the scroll position, violating the invisibility guarantee. True eviction requires index-stable windowed faulting, which IS M2's data layer; building it without M2's SQLite index would be M2's grid work without its foundation. | (deferred with M0.4 to M2, where faulting makes it actually invisible) |
| M0.3 ✅ shipped 2026-06-11 | **LRU cap on the disk image cache** | `ImageCache` tracks access times (file `contentModificationDate` touch on disk hit) and trims to a budget (default 512 MB) on launch + app-backgrounding (iOS exposes no public low-disk notification — deviation from the original "low-disk" trigger). | Cache-only; worst case is a re-download. |
| M0.4 ↪ **folded into M2** (2026-06-11) | **Grid tolerates gaps** | If eviction creates a gap between the resident window and a fast fling destination, `loadMoreLibrary` fetches the page containing the first visible index rather than strictly `highest+1`. | (deferred with M0.2 — gaps only exist once eviction does) |
| M0.5 ✅ shipped 2026-06-11 (with M0.1) | **Telemetry log line** | One `NSLog` per persist reporting media counts (resident / persisted; no client-side server total exists), so scale problems show up in console before users feel them. | Log-only. |

**Exit criteria:** instrument a simulated 50k-item library (stub API): persist encode time flat vs. today's 100-item builds; resident memory flat while scrolling 5k items; capture diff clean.

**Step M0.1 ships immediately even if M0.2–M0.4 wait** — it alone removes the unbounded-persist hazard.

## Phase M1 — Server: cursor (keyset) pagination

> Offset paging is the only piece that degrades server-side no matter what the client does. Effort: ~1–2 sessions. Backward compatible — `page/limit` keeps working during migration.

> **✅ SHIPPED 2026-06-11** (M1.1–M1.6). Notes vs. the spec: (a) a redundant
> `createdAt <= cursor` bound rides alongside the OR keyset predicate — Prisma's OR form
> alone is only a post-scan filter and deep pages degraded O(depth); the bound lands in the
> index condition (EXPLAIN-verified). (b) M1.4 solved by *omitting* `total` on cursor
> responses instead of estimating — clients keep the exact total from their initial page-1
> response, so no count(*) runs on the deep-paging path at all; page-mode counts unchanged
> for the web client. (c) Exit criteria: walk-equivalence + filter parity + plan-shape
> verified against local data and 8 vitest tests; the 250k-row p95 benchmark has NOT been
> run — do it before declaring victory at real scale. iPhone adoption (M1.5) shipped
> alongside; web client stays on page mode by design.

| # | Step | Detail |
|---|---|---|
| M1.1 | **Composite index** | `@@index([organizationId, createdAt(sort: Desc), id])` on `Media` (schema YAML → Prisma migration). Keyset queries become index-range scans at any depth. |
| M1.2 | **Cursor variant of `getLibrary`** | Accept `cursor` (opaque base64 of `createdAt|id`). Where clause: `(createdAt, id) < (cursorCreatedAt, cursorId)` — Prisma: `OR [{createdAt: {lt}}, {createdAt: eq, id: {lt}}]` — same `orderBy: [{createdAt: 'desc'}, {id: 'desc'}]`. Return `nextCursor` + `hasMore`. Keep `total` via a separate cached/approximate count (exact `count(*)` at millions of rows is itself slow — see M1.4). |
| M1.3 | **Route + OpenAPI/Swagger update** | `GET /api/organizations/:orgId/media/library?cursor=...&limit=100`. `page` continues to work (deprecated) so the web client (`/client` admin media views) is unaffected until it migrates. **Cross-app check (impact table): client + iphone both consume this endpoint.** |
| M1.4 | **Scalable counting** | Replace per-request `prisma.media.count` with: exact count below a threshold (e.g. 10k), else Postgres planner estimate (`pg_class.reltuples`-based) or a cached count refreshed on upload/delete. The UI only uses `total` for "load more?" decisions and a label — estimate-grade is fine; `hasMore` from keyset is exact. |
| M1.5 | **iPhone client adoption** | `MediaActions` stores `mediaLibraryNextCursor` (in AppState, not persisted); `loadMoreLibrary` sends `cursor` instead of `page`. Filter changes reset the cursor. Falls back to page-mode if the server doesn't return a cursor (deploy-order safety). |
| M1.6 | **Search/filter parity** | Cursor path must accept the same `type/tags/leaders/q` filters; keyset predicate composes with the existing `where`. Server tests for: stable iteration under concurrent inserts, filter + cursor, tag-join + cursor. |

**Exit criteria:** seeded 250k-row media table; p95 page fetch flat from page 1 to page 2,500; server tests green; both clients function against the deployed endpoint (web on `page`, iPhone on `cursor`).

## Phase M2 — Local media index + windowed faulting (the Photos data layer)

> The architectural core. Invisible: same grid, same cells, same transitions — backed by a database instead of an array. Effort: ~2–3 weeks. Depends on M1 (cursor sync) and structural Phase 2 (done — @MainActor state, DI, coalescing).

| # | Step | Detail |
|---|---|---|
| M2.1 | **Local store: SQLite via GRDB** (recommended over Core Data: value-type friendly, plays well with `@Observable`, trivial to test) | New `MediaStore` module: `media_index` table mirroring list-level fields only (id, type, title, createdAt, thumbnailUrl, duration, usageCount, tag ids, leader id). Detail-level data stays API-fetched on demand (as today via `loadDetail`). |
| M2.2 | **Incremental sync engine** | `MediaSyncEngine` pulls by cursor: initial backfill (newest-first, resumable, background-priority after first screen) + delta sync (since `lastSyncedAt`, plus tombstones for deletes — requires server `updatedAt >= ?` + `deletedAt` soft-delete or a changes feed; small M1 addendum if soft-deletes don't exist yet). Upload/delete/tag mutations write through to both API and local index. |
| M2.3 | **Windowed data source** | `MediaWindow` (@MainActor, @Observable): exposes `totalCount`, `item(at:) -> MediaLibraryItem?` (faults from SQLite in batches of ~500 around the viewport, LRU-evicts distant batches), and change notifications. The diffable snapshot is replaced by count-based sections with cell reconfiguration on fault completion — at 1M items the grid binds to a count, never an id array. |
| M2.4 | **Sort/filter in SQL** | `filteredMedia` (MainLibrary) and picker queries become indexed SQL (`ORDER BY created_at DESC`, `usage_count DESC`, `title COLLATE NOCASE`; tag/type/leader/time-cutoff predicates). The per-render array sort is deleted. Search: SQLite FTS5 over title/description/alt-text for instant local search; server semantic search stays for the "smart" path. |
| M2.5 | **AppState integration** | `AppState.mediaLibrary` (EntityStore) is retired for the library surface; `mediaLibraryTotal` comes from the index. Media leaves `PersistedState` entirely (the index file *is* the persistence). MediaLibraryPicker + VideoActivityManager move to `MediaWindow` queries. Capture fixtures seed the index (CaptureEnvironment writes rows instead of upserting the store). |
| M2.6 | **Thumbnail tiers** | Grid switches to `-thumb` (150px) **only at the future 5+ column zoom level**; 3-column stays `-md` (the audit's correction stands). `ImageCache` budget tuning: count-based limit scales with tier. Prefetch window widens (cheap thumbs). |
| M2.7 | **Scrubber readiness** | Because the full index is local, jump-to-anywhere becomes O(1): expose `MediaWindow.item(at:)` faulting + month boundaries query (`SELECT strftime('%Y-%m', created_at), count(*) GROUP BY 1`) for the future date scrubber. No UI yet (M3). |

**Exit criteria:** 250k-item seeded index; cold launch to first grid frame < 500ms; fling from top to bottom faults smoothly with no main-thread stalls > 16ms (Instruments); memory flat (< 150MB) regardless of scroll depth; airplane-mode relaunch shows the full previously-synced library; capture diff clean.

**Sequencing note:** M2 should land **before** the audit's media redesign (pinch zoom, pager, date sections) — the redesign's interactions assume this data layer. Recommended order relative to the structural plan: structural Phase 3 (navigation) → M0 already done → M1 (server, parallelizable with Phase 3) → M2 → M3.

## Phase M3 — The visible redesign (design-led, separate approval)

> Listed for completeness; this is the audit's deferred "media Phases 2–3" plus scale-specific UI. Every item is a visible change requiring design sign-off — nothing here is structural.

- Pinch-to-zoom column levels (3 ↔ 5 ↔ 7+), driving thumbnail tier selection (M2.6)
- Date/month section headers + fast scrubber (powered by M2.7 month index)

- Upload pipeline UX for bulk imports (progress, background upload queue)

## Decision points / open questions

- **A — Soft deletes / changes feed (M2.2):** delta sync needs the server to expose deletions. Soft-delete column vs. an `media_changes` log table — server-side decision, small but schema-touching.
- **B — GRDB dependency:** first third-party dependency in the iPhone app (currently SPM has SnapshotTesting only in the test target). Alternative is Core Data (no new dependency, more ceremony). Default recommendation: GRDB.
- **C — Approximate totals (M1.4):** product call — is "~12,400 items" acceptable in the UI when exact counting becomes expensive? (Recommended: yes.)
- **D — Web client parity:** the Laravel/Vue admin media views inherit the same offset-paging limits; migrating them to cursor is a separate client-repo task once M1 ships.

## Verification protocol

Same as the structural plan: capture-diff per phase (media fixtures should be activated from `_media/` pending set when M2 starts), characterization tests for `MediaWindow` faulting/eviction and sync-engine resumability, one commit per step, `/transition-review` on anything touching the grid↔overlay transition. Scale exit criteria run against a seeded-fixture API stub (50k–250k rows) so they're reproducible in CI.
