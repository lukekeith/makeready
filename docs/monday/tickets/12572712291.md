# Image optimization — WebP/AVIF output + right-sized delivery

## Problem
Follow-up Feature spun out of the "image optimize" question ([12271625826](12271625826.md)). The current pipeline is optimized-but-not-ideal: uploads are resized via `sharp` to a fixed 3-size **JPEG** ladder (1200/400/150 px · q85/80/75) on Cloudflare R2, served as-is with no transform. This feature closes the two highest-leverage gaps:
1. **Modern formats** — add WebP (optionally AVIF) output. ~25–50% smaller at equal quality.
2. **Right-sized delivery** — the coarse ladder mis-fits surfaces (the Program Home hero cover loads the 400px `-md` full-bleed and looks soft). Serve each surface the size it needs.

"Done" looks like: images ship as WebP/AVIF to clients that accept them, and each surface receives an appropriately-sized image, with no regression to the existing JPEG fallbacks.

## Open design decision (resolve BEFORE building — this is why affected_areas is provisional)
**More static variants vs. an on-the-fly transform layer.**
- **Option A — extend the static pipeline:** add WebP/AVIF outputs (and maybe a larger hero size) to `sharp` + `uploadImageVariants`; consumers pick format/size by URL convention. Simple, no new infra; more R2 objects per image, still a fixed ladder.
- **Option B — transform layer** (Cloudflare Images or R2 image resizing): store one high-quality master, transform per request (arbitrary size + `Accept`-based format negotiation). Fewer stored objects, ideal fit, best format per client; new infra + cost + delivery-path change.
Pick one before `/monday-resolve` — the affected surfaces differ, and the client-side change (Option B lets clients drop hard-coded `-md`/`-thumb` URL munging) hinges on it.

## Affected areas
> `provisional` — real surfaces identified and verified, but the in-scope client set depends on the design decision above. Re-run /monday-ticket to `confirmed` once Option A vs. B is chosen.

### In scope (server — both options)
- **Ingest/resize:** `server/src/routes/media.ts:1206-1215`, `server/src/routes/programs.ts:3658-3663`, `server/src/routes/events.ts`, `server/src/routes/groups.ts`, `server/src/routes/members.ts:1357` (avatar) — the `sharp` pipelines that would gain WebP/AVIF output and/or feed the transform layer.
- **Storage:** `server/src/services/storage.ts:194` (`uploadImageVariants`) — variant writer; new formats/sizes land here (Option A) or it simplifies to a single master (Option B).
- **Transparency handling:** the force-to-JPEG conversion in the same paths — decide per-format (WebP/AVIF preserve alpha; a PNG/logo currently gets matted).

### In scope (clients — depends on decision)
- iPhone variant helper `iphone/MakeReady/Utilities/String+ImageVariants.swift` (+ `ImageCache.swift`) and cover/card consumers (e.g. `ProgramHomePage.swift:755` uses `.mediumImageUrl`). Option B would let these stop hard-coding `-md`/`-thumb`.
- Web consumers reading `coverImageUrl` / `thumbnailUrl` (e.g. `client/resources/js/islands/leader-app/stores/leader-library.store.ts`, card components).

### Explicitly out of scope
- **Video** (Cloudflare Stream) — separate pipeline.
- The lesson-player member surfaces' *layout* — this is about the bytes/format they load, not their CSS.
- Any change to what users upload — sizing/format stays server-enforced.

### Cross-app impact
Per the root CLAUDE.md table: this is a **media pipeline** change touching the server + BOTH consumers (iPhone + web) — new image URLs/formats must be consumed by both. If Option B changes the delivery host/URL shape, every image-loading surface in both clients is affected; stage it behind the existing JPEG URLs as a fallback.
