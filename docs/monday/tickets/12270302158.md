# rescheduling

## Problem
Original report: after publishing/enrolling a study, the leader could never get back to the publish/enrollment screen to republish or change the enrollment. The *republish* half was addressed (Program Home's published badge re-opens the publish/draft dialog). The remaining, now-primary ask is a full **edit-enrollment capability** that does not exist yet.

**The fix the user has designed (this is the scope contract):**
1. **Tapping an enrollment shows an action-button modal** (the same `LessonActionMenu` used on the home dashboard when tapping a lesson) with two options — **Edit lessons** and **Edit enrollment** — instead of immediately loading the lesson-schedule editor.
2. **"Edit enrollment"** opens the "Confirm enrollment" screen repurposed as an **"Edit enrollment"** page, with these changes:
   - Title "Confirm enrollment" → **"Edit enrollment"**.
   - Tapping the **schedule section** (day-of-week strip) → slides to the **"Select dates"** calendar. All drilldowns on this page slide in **from the right** (Edit enrollment is the *first/root* page, unlike create where Confirm is the *last* page and drilldowns come from the left — same HStack+offset mechanism, reversed panel order).
   - Tapping the **group name** → slides to the **group-select** page.
   - Tapping the **study-program name** → slides to the **study-select** page. Changing the study reschedules the enrollment: the only guarantee is that **enrolled lessons with zero activity are removed**; new enrollment lessons are created from the new program starting at the enrollment's start date (the edited start date if changed, else the original).
   - Left nav: back chevron → **"Cancel"** link (dismiss, save nothing).
   - Right nav: **"Save"** (currently "Confirm").
   - A **warning banner** at the top summarizes any destructive / lesson-impacting changes.
   - **Save** → a `DialogOverlay` confirmation that either warns (if destructive) or says "This will update the existing enrollment to match your changes."

"Fixed" observably looks like: tap an enrollment → action menu → Edit enrollment → change group/study/schedule/settings → Save → confirmation → the enrollment (and members' calendars) reflect the edits.

### Design decisions (2026-07-21, from user's spec + implied requirements)
These are load-bearing and part of the confirmed scope:
1. **Atomic staging** — "Cancel ensures nothing saved" means the entire edit (group/study/schedule/settings) is held in client state and committed **only on Save**, in **one server transaction**. Study-swap/reschedule must NOT be applied the moment a sub-value is picked.
2. **Impact preview** — the warning banner and the "warn if necessary" Save-dialog require a **server dry-run/impact-preview** (which lessons would be removed, which dates shift). The client renders the banner from that preview; the final Save commits.
3. **2-item menu variant** — the enrollment-tap menu shows only **"Edit lessons"** + **"Edit enrollment"** (a subset/config of `LessonActionMenu`'s ~6 fixed rows). "Edit lessons" → existing `EnrollmentSchedulePage`; "Edit enrollment" → the new flow. (Label "Edit lessons", not the existing "Edit Activities" row text.)
4. **Edit-mode sub-page nav** — in the edit flow the drilldowns (`SelectEnrollDatePage`, group/study select) **apply-and-return** to "Edit enrollment" (nav label "Done"), not the wizard-advance "Next".
5. **Members section** — "Members included/excluded" reflects the current (possibly changed) group; **read-only** in the edit flow (not a drilldown).

## Root cause
Not a defect — a missing capability. The enroll flow is create-only: `ConfirmEnrollmentPage` and the `EnrollmentFlowModal` wizard have no edit entry point, tapping an enrollment jumps straight to lesson editing, and the server exposes no endpoint to change an enrollment's group/study/start-date/schedule (only four scalar PATCH fields). The reschedule + study-swap semantics the design calls for do not exist server-side and must be built.
