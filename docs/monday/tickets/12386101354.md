## Problem
Two member-web reports on the same program ("Struggle Well", Day 1), each with a screenshot:

1. **"Navigation covering the verse" (exegesis)** — in the exegesis step the top of the verse is flush against / under the lesson header. See Root cause.

"Fixed" observably: (1) verse top clears the header (done); (2) a reading either shows its content or, when it genuinely has none, shows an explicit empty-state / auto-advances instead of a blank slide.

## Root cause
- **Sub-1:** `exegesis-step.vue` static 120px top inset < live header height (matches ticket 12415662995). **Fixed `fbcd848`.**
- **Sub-2 (provisional):** the member read step has no empty/blank-slide guard. `buildPayload` drops empty (or empty-after-live-resolution) blocks and, if none remain (or the active slide has no visible content), `ActivityPreviewPlayer` parks on a blank frame while still reporting "Reading complete". The 3 block-markers in the screenshot indicate blocks WERE present, so the leading hypotheses are: (a) a scripture block whose live `/api/bible` resolution returned empty for that reader; or (b) a render/positioning bug that left an otherwise-populated slide visually blank. Distinguishing them needs the actual "Struggle Well" Day-1 read activity, which isn't in the local DB. The review's CF-19 "reorder-then-enroll → no lessons" theory is contradicted by the screenshot (the lesson exists and is being read).
