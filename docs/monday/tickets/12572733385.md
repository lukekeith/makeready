# iPhone exegesis editor — "Couldn't save changes"

> **Split 2026-07-19** from "validation error" ([12268645785](12268645785.md)), which had bundled this unrelated iPhone bug with a member web SMS-verification issue.

## Problem
On the iPhone native **Edit Exegesis Activity** editor, saving fails with a red **"Couldn't save changes"** banner after the user has built up highlights/notes ("did a bunch of work"). Root cause is now identified from code: **the client and server disagree on highlight-overlap semantics.** `saveTitle()` → `savePendingNotes()` decides create-vs-update by an **exact-range match** (`matchingExegesisHighlight(for: range)`); when a note's range overlaps but doesn't exactly equal an existing highlight, the client tries to **create** it, and the server **rejects any overlap** with a 400 — which throws out of the whole save and surfaces the generic banner. (Secondary trigger: after a passage change the server deletes the block + all highlights, so stale `readBlockId`/`highlightId` values produce 404s on the next save.) "Fixed" = overlapping/adjacent highlight edits save (via merge/update, or a guarded UI), and a passage change doesn't leave the editor saving against dead IDs.

## Root cause
Client/server highlight-overlap mismatch: `savePendingNotes` (`EditExegesisActivityPage.swift:452`) only recognizes an existing highlight on an **exact** range match, so an overlapping edit is sent as a **create**, which the server rejects (`programs.ts:3034-3036`, 400). The thrown error aborts the entire `saveTitle` and shows "Couldn't save changes" (`:779`). Secondary: stale `readBlockId`/`highlightId` after a passage change (server deletes them, `programs.ts:2741-2744`) → 404s on save.
