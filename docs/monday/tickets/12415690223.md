# member - error

## Problem
A study program that was enrolled, unenrolled, edited, re-enrolled, and finally
**deleted** keeps rendering in both the **member group home** and the **Group
Leader (GL)** UI, and can't be dismissed. Root cause is server-side: `DELETE
/programs/:id` is a *soft* delete (`isActive: false`) that leaves the enrollment +
lesson schedules intact, and **none** of the group-scoped enrollment endpoints
filter on the program's active state. So every list keeps returning the dead
study's enrollment. "Fixed" observably: deleting a study removes it from every
member and leader enrollment list on the next load (no dead card left to error on).

## Reports (verbatim)
- (2026-07-01) "This is a study that was unenrolled and then edited and enrolled and deleted - I can't make this go away from my member UI or the GL UI"

### In scope — Server (three group-scoped enrollment queries missing the program-active filter)
- **`server/src/routes/enrollments.ts:716`** — `GET /groups/:groupId/study-enrollment`
  (singular; `enrollment.findFirst({ where: { groupId }, orderBy: createdAt desc })`) —
  the member "current study" lookup. A soft-deleted study can be the "most recent".
- **`server/src/routes/enrollments.ts:893`** — `GET /groups/:groupId/study-enrollments`
  (plural; `enrollment.findMany({ where: { groupId } })`) — **the screenshot**: the
  member group-home "Your studies" list (`GroupHomeController` → `MemberStudiesIsland`).
- **`server/src/routes/enrollments.ts:1187`** — `GET /groups/:groupId/enrollments`
  (`enrollment.findMany({ where: { groupId } })`) — the leader/GL enrollments list
  (consumed by the iPhone leader app `EnrollmentActions.loadEnrollments(groupId:)` and
  the web LeaderApp). The `isActive: true` already present here is on the *group*
  lookup, not the enrollment/program.
- **Fix:** add the relation filter `studyProgram: { isActive: true }` to each `where`
  (i.e. `where: { groupId, studyProgram: { isActive: true } }`). Optionally also
  `select`/`include` `isActive` so clients could filter defensively.
- **Endpoints:** the three above. `DELETE /api/programs/:id` (`programs.ts:1402-1417`,
  soft delete) stays as-is — keeping member progress on soft-delete is intentional;
  the fix is the read filter, not the delete.

## Root cause
`DELETE /programs/:id` soft-deletes (`programs.ts:1417`, `data: { isActive: false }`),
leaving the enrollment and its lesson schedules in place. The three group-scoped
enrollment endpoints (`enrollments.ts:716`, `:893`, `:1187`) each query
`enrollment.find*({ where: { groupId } })` with **no** `studyProgram.isActive`
condition — so they return enrollments whose program is soft-deleted. Result: the dead
study renders in the member group home AND the GL enrollments list, with no way to
dismiss it. (CF-4 in [code-fixes.md](../code-fixes.md).)

## Verification
- **Code-level:** `enrollments.ts:716`, `:893`, `:1187` — each `where: { groupId }` has
  no `studyProgram.isActive` filter (confirmed 2026-07-20); `programs.ts:1417` is a soft
  delete.
- **Manual repro (local, web member + leader):** enroll a group in a study → delete the
  study as its creator → reload the member group home (`/group/...`) and the leader
  group/enrollments screen → the dead study still lists. After fix: gone on reload.
