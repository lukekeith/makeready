# member - error

## Problem
A study that was unenrolled, then edited, re-enrolled, and finally **deleted** keeps rendering in both the member UI and the Group Leader UI, and cannot be dismissed. Two compounding causes: study delete is a *soft* delete that touches nothing else, and none of the enrollment-list endpoints filter on the program's active state (CF-4) — so both clients keep receiving (and both also cache) the dead study. "Fixed" observably: deleting a study removes it from every member and leader list immediately (or the lists exclude inactive programs), with no dead card left to error on.

## Reports (verbatim)

## Root cause
Per CF-4 ([code-fixes.md](../code-fixes.md)): study delete is a soft delete (`programs.ts:1197-1200`) and **none** of the enrollment-list endpoints filter on the program's active state — the leader web store even carries a no-op guard for behavior never implemented server-side (`leader-group-home.store.ts:357-359`). Client-side caching (CF-7: `EnrollmentActions.swift:95-128` cache-first on iPhone; `enrollments.domain.ts:70-83` refetch short-circuit on web) keeps the ghost alive even across the fix boundary. Fix: add `studyProgram: { isActive: true }` to the four list endpoints' `where`, and/or cascade enrollment removal on soft delete.
