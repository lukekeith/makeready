## Answer (ready to post on the ticket)
> Images are **not** stored at full resolution. Every upload runs through `sharp` on the server before hitting storage: the "original" we keep is re-encoded to **max 1200px on the long edge, JPEG quality 85**, and we also generate a **400px** (`-md`) and **150px** (`-thumb`) variant. A 4000px photo you upload is stored as ~1200px. Covers, media-library photos, and group/event images all follow this; avatars are capped at 512×512. Files live on Cloudflare R2 (`images.makeready.org`) and are served as-is. So yes — they're optimized, not raw. Two honest caveats if we want to go further: (1) we only output JPEG — adding WebP/AVIF would cut ~25–50% more at the same quality; and (2) it's a fixed 3-size ladder with no on-the-fly resizing, so some screens don't get an ideal fit (e.g. the study-cover header currently loads the 400px variant and looks soft on large phones). Both are enhancements, not bugs.

## Assessment — optimized, but not ideal (verified 2026-07-19)
"Optimized" (a real resize pipeline exists) ≠ "ideal." Ranked improvement levers:
1. **Format** — JPEG-only (q85/80/75). No WebP/AVIF. Biggest single win (~25–50% smaller at equal quality); `sharp` supports it trivially. → adding it is a **Feature**, server-side.
2. **Coarse fixed ladder + wrong-fit selection** — only 1200/400/150 exist, and consumers pick imperfectly: the Program Home hero cover pulls the **400px `-md`** (`ProgramHomePage.swift:755`) and upsizes it full-bleed → soft. A transform layer (Cloudflare Images / R2 resizing) would let each surface request the exact size + negotiate format. → **Feature**, spans server + both clients.
3. **Force-to-JPEG drops transparency** — every upload is converted to JPEG, so a PNG/logo with alpha gets a matte. Fine for photos, wrong for graphics. → potential **Bug** if transparent uploads occur.
4. **1200px largest cap** — fine for cards, borderline for a full-screen cover on large/retina devices (~1290 physical px). No DPR-aware hero size.
5. **Encoder** — plain `sharp` JPEG; enabling mozjpeg/progressive is a minor extra few %.

None of these are in scope for *this* ticket (a question). If you want to pursue #1/#2, that's a new Feature work item — re-run /monday-ticket to scope it.
