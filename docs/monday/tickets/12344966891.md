# UI not updating

## Problem
On the **iPhone app**, after a leader unenrolls a group from a study program, the
enrollment card keeps showing on the **Groups → Enrolled** tab, and acting on that
ghost card produces an "Enrollment not found" error. Two sub-issues, one root cause:

1. **Ghost enrollment card.** The unenrolled "Struggle Well / Stickane Family" card
   still renders in the Enrolled list. **Bug — STILL RELEVANT.**
2. **Error on the ghost card.** Selecting / swipe-unenrolling it opens the "Unenroll"
   modal, which 404s on the already-deleted enrollment → "Enrollment not found /
   Try Again". **Bug — STILL RELEVANT** (a downstream symptom of #1).

Both attachments are the iPhone native app (6-tab NavBar, no browser chrome). The
server delete cascades cleanly — this is **purely a client-side cache bug**: the
unenroll mutation prunes the wrong stores, so the Enrolled tab keeps rendering a
deleted enrollment until a manual pull-to-refresh.

"Fixed" observably: unenrolling removes the card from the Enrolled tab immediately,
and a lingering card never produces a user-facing error.

## Root cause
**Dual-store cache gap on iPhone.** Enrollments are indexed in two parallel stores:
- `enrollments` + `groupEnrollmentIndex` + `programEnrollmentIndex` (written by `loadEnrollments(groupId:)`, pruned by `deleteEnrollment`), and
- `programEnrollments` + `programProgramEnrollmentIndex` (written by `ProgramActions.fetchProgramEnrollments`, which correctly REPLACES its index).

`MemberHomePage`'s Enrolled tab reads the **second** store; `deleteEnrollment` prunes
only the **first**. So an unenroll leaves the card in the tab's read-store. It would
self-heal via `getProgramEnrollments`' background refresh (which `.replace`s the
index) — but `loadAllEnrollments` only calls that when `!hasCached`, and the stale
entry keeps `hasCached == true`; worse, `deleteEnrollment` decrements
`program._count.enrollments`, so once it hits 0 the tab's `hasEnrollments` gate skips
the program entirely and the ghost never refreshes away. Result: the card persists
until manual pull-to-refresh, and acting on it 404s (server already deleted it) →
`UnenrollOptionsModal` error phase ("Enrollment not found" was the server message;
that literal string is no longer hardcoded in the app, but the dynamic-message error
path remains).
