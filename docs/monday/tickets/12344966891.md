# UI not updating

## Problem
After unenrolling a group from a study, the enrollment still shows up in the UI, and tapping the lingering card throws an error. The server delete cascades cleanly — the ghost is client-side stale cache: the iPhone renders disk-cached enrollment lists cache-first and the web store short-circuits refetch, so a deleted enrollment keeps rendering until something 404s on it. A related server gap (CF-4: list endpoints don't filter soft-deleted programs) feeds the same "items remain after a change" class. "Fixed" looks like: unenrolling removes the card everywhere immediately, and a stale card never produces a user-facing error.

## Root cause
Per CF-7 ([code-fixes.md](../code-fixes.md)): server delete cascades cleanly, but the iPhone persists enrollment lists to disk cache-first (`EnrollmentActions.swift:95-128`) and the web store short-circuits refetch via `loadedGroupIds` (`enrollments.domain.ts:70-83`). A stale card then 404s on `GET /enrollments/:id` (`enrollments.ts:677`) or on `completion-stats` — the one endpoint that filters soft-deleted programs (`:1646-1656`) — producing the visible error. 2026-07-19 note: reconcile with CF-4's missing `isActive` filters when fixing.
