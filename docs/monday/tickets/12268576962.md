# calendar breaks when unpublishing

## Problem
After publish → enroll → unpublish → re-enroll, days the leader adds to the study never showed up for the member. At report time, a day-count increase (or per-day add) created new live `Lesson` rows on the study, but the member renders from **per-enrollment snapshot rows** (`LessonSchedule`), which nothing back-filled — so the added days were invisible on the member calendar. "Fixed" looks like: a lesson added after enrollment reaches the enrolled group's calendar.

**Current state (nuanced): the structural defect is fixed, but the reported symptom still reproduces under default config.** The study-sync version/propagation system now provides a working path (leader "Publish updates" → cut version → enrollment sync `applyVersion` creates the missing schedules + calendar events). BUT that path is gated behind **two manual leader steps** and the default is off: (1) adding a day never auto-publishes — the leader must hit "Publish updates" to cut a version; and (2) the enrollment must be `syncMode: AUTO` (default is **`OFF`**) or the leader must explicitly apply the pending sync. A leader who follows the reporter's exact steps (add days, do nothing else) will therefore still see the days fail to appear. The remaining work is a **product/default decision**, not a broken code path.

## Root cause
The member calendar renders from per-enrollment `LessonSchedule` snapshots, not the live curriculum. Adding a day creates a live `Lesson` row (`programs.ts:3800` / `:933`) but does **not** create matching `LessonSchedule` rows for existing enrollments — so at report time there was no path from "add day" to "member sees it". The study-sync rework closed the gap: leaders now cut a program version ("Publish updates") and enrollments sync to it, with `applyVersion` explicitly creating schedules + events for added lessons.

## Residual gap (needs a product decision — this is what /monday-resolve would act on)
Propagation is now **leader-initiated**, not automatic on lesson-add. For an added day to reach the member, BOTH must happen: the leader hits "Publish updates" (cuts a version), AND the enrollment is `syncMode: AUTO` **or** the leader applies the pending update. Default `syncMode` is `OFF` (schema.prisma:983, enrollments.ts:258), so the reporter's literal steps still fail silently (the leader only gets a notification, if anything). The structural defect — *no path at all* — is resolved, but closing this to the reporter's satisfaction requires one of:
- **(a) Default new enrollments to `AUTO`** — smallest change (enrollments.ts:258 / schema default); published updates then flow automatically. Risk: leaders lose the review gate; existing OFF enrollments unaffected.
- **(b) Auto-publish + fan-out on lesson add** — add-lesson / day-increase endpoints call `publishProgramVersion` after mutating. Larger; changes publish semantics.
- **(c) Accept as by-design** and rely on the "updates available" notification + leader UX to surface "Publish updates". Then this closes with no code change (verify the leader UI makes the affordance discoverable).
