## Audit Criteria

Each audit in this series evaluates the same eight criteria so scores are comparable over time:

1. **State architecture fidelity** — Does the code follow the documented AppState + Actions + EntityStore pattern? Are violations growing or shrinking?
2. **Thread/concurrency safety** — Is state mutation compiler-enforced (`@MainActor`, actors) or convention-enforced?
3. **Navigation & transition robustness** — Is there a single navigation system with typed routes, or N hand-rolled re-implementations? Are transitions completion-driven or timer-driven?
4. **Error handling & observability** — Are errors surfaced to users? Is logging structured, leveled, and privacy-safe?
5. **Security posture** — Credential storage, transport security, sensitive data in logs, test surface in production builds.
6. **Design system adoption** — Token usage (color/typography/motion) vs. inline hardcoded values; Dynamic Type; accessibility.
7. **Testability & coverage** — Dependency injection, unit tests on the State layer, decoding fixtures.
8. **Documentation accuracy** — Does `.claude/CLAUDE.md` (and the scar-tissue docs) match reality? Drift compounds because AI sessions inherit it.

## 1. Architecture & State Management

### What's genuinely good (keep it)

- **Normalized entity stores + relationship indexes** (`State/EntityStore.swift`, `State/RelationshipIndex.swift`) — clean generics, O(1) lookups, no duplicated parent/child data. This is the right shape; most teams reinvent it badly.
- **Cache-first loading with disk persistence** — instant UI on launch, corrupted-file self-healing, past-event pruning on restore.
- **Tri-state loading model** (`loading` / `refreshing` / `error` in `LoadingStateManager`) — exactly what cache-first UIs need.
- **Actions as stateless `@MainActor` structs** with a documented single write path (`AppState.mutateScheduledActivity`).
- **Unified `APIError`** with Zod-error parsing, EXIF stripping on upload, environment-driven base URL.

## 2. Navigation & Transitions (the #1 pain point)

### Diagnosis: why transitions keep breaking

**The app has no navigation system — it has ~10 independent re-implementations of one.** `NavigationLink` and `navigationDestination` appear **zero** times. Page movement is hand-rolled `HStack` + `offset(x:)` sliding rebuilt per page with *different* state shapes (Int index in `GroupHomePage`, single Bool in `EditDay`, two Bools + derive function in `CreateProgramPage`), plus `OverlayManager` for modals/menus, plus escape hatches (24 `.fullScreenCover`, 12 `.sheet`, 5 `NavigationStack` islands, 5 deprecated `NavigationView`s). Five transition engines run concurrently with different curves, z-handling, and gesture semantics.

The 10 failure classes documented in `SWIFTUI_TRANSITIONS.md` / `SWIFTUI_ANIMATION_PATTERNS.md` are mostly **one disease**: *structural identity changes during a manually-animated offset transaction*. When you animate `offset(x:)` yourself, anything that changes view structure mid-flight (lazy item realization, `if let` flipping when async data arrives, un-laid-out text) is inserted outside the transaction and lands at its final position. SwiftUI's built-in containers coordinate insertion + animation atomically; this codebase re-derives that coordination by ritual (two-step pattern, deferred onAppear, init-time cache priming, VStack-not-LazyVStack) — and every new page must remember all the rituals. The institutional knowledge lives in markdown, not code.

### Key evidence

- **Same "push to detail" transition, four implementations**: `GroupHomePage.swift:66,120` (Int index, explicit `withAnimation`, screen 0 to the *left*), `ProgramHomePage.swift:160-187` (Bool offset + implicit `.animation` + a nested ZStack *opacity cross-fade* as a third mechanism), `CreateProgramPage.swift:146-160` (two Bools, **two** `.animation` modifiers watching different values — the exact anti-pattern the docs warn about), `MemberHomePage.swift:493` (the only caller of `overlayManager.presentPage`, the one centralized push mechanism, used once in the entire app).
- **Time-based choreography**: 65 `DispatchQueue.main.asyncAfter` calls — 17 hardcoding the magic `0.35` "wait for dismiss" delay that must mentally match `ModalAnimations` durations. iOS 17's `withAnimation { } completion: { }` is available and **unused**.
- **State sprawl**: ~106 presentation booleans across pages; zero navigation state in AppState; no route type, so deep links are imperative pokes (`MainView.swift:164-201`) and illegal states (`showEditDay && !showProgramHome`) are representable.
- **MODAL_GUIDE violated by its own codebase**: `GroupHomePage.swift:140` and `EnrollmentSchedulePage.swift:102` present forms via `fullScreenCover` — exactly what the guide forbids.
- **Eager `AnyView` capture**: `OverlayManager.swift:187,254` builds modal content at present-time and erases it, defeating structural diffing — a contributor to the async-content-pop class.
- **Inconsistent motion tokens**: durations 0.2×49 / 0.3×40 / 0.25×18; `easeInOut`×89 vs springs ×24; push uses `easeOut(0.3)`/`easeIn(0.25)` while modals use springs — no shared vocabulary.

### Target architecture (recommended)

**Formalize the existing custom system around a typed route model — do not migrate wholesale to NavigationStack.** The design language (bottom-sheet modals, in-modal horizontal sliders, custom scrims) doesn't map to stock navigation, the app is iOS 17+ (no `.navigationTransition` customization until iOS 18), and 239 page files make a rewrite the riskier path. OverlayManager's core is sound; what's missing is the layer above and below it:

1. **`Route` enum + `NavigationCoordinator` (@Observable)** — typed cases (`.groupHome(groupId:)`, `.editDay(programId:lessonId:)`) replace stringly-typed `OverlayID`; coordinator owns a route stack per overlay layer with `push/pop/present/dismissAll`; deep links become `coordinator.navigate(to:)` — declarative and testable.
2. **One canonical `SlideStack` container** — extract the HStack+offset pattern *once*: encapsulates two-step insertion internally, owns the edge-swipe-back gesture (currently absent from all sliders), applies `.compositingGroup()` and a single `.animation`, exposes completion callbacks. Delete the 10 hand-rolled copies.
3. **`Motion` token enum** — `Motion.push`, `Motion.modal`, `Motion.micro`; hardcoded durations/curves become lint failures.
4. **Completion-based sequencing** — replace all `asyncAfter(0.35)` choreography with iOS 17 animation completion handlers and `dismiss(then:)` coordinator helpers.
5. **Lazy overlay content** — `OverlayItem` stores `() -> AnyView` so content builds at render time, not present time.

**Migration steps (each independently shippable):**
1. `Motion` tokens + completion helpers; mechanically replace the 17 `0.35` waits. *(low risk, immediate payoff)*
2. Build `SlideStack`; migrate `EditDay` first, verify against the failure-class checklist, then the remaining 9 sliders.
3. Introduce `Route`; key OverlayManager by route; convert deep links.
4. `NavigationCoordinator` owns the route stack; cut the 106 booleans to <30 genuinely-local toggles.
5. Sweep escape hatches: the 2 forbidden fullScreenCovers → `presentModal`; replace 5 `NavigationView`s; delete the unused hello-world `ContentView.swift`.

## 3. Media Library UI vs. Photos-App Target

### Current state — better foundation than expected

The main grid (`Pages/Media/MediaLibraryGrid.swift`) is already a **UICollectionView with real cell reuse** — edge-to-edge 3-column, 2pt gutters, square cells, per-cell cancellable thumbnail tasks, two-tier `ImageCache` (NSCache + disk, actor-based). The detail view (`Pages/Media/MediaDetailOverlay.swift`, pure UIKit) has a hand-rolled **zoom-from-cell expand with interactive swipe-down dismiss**, pinch 1–4x, progressive thumbnail→full-res loading, and inline video playback. That's roughly half the Photos experience already built, on the right technology (UIKit collection view outperforms `LazyVGrid` at scale and gets two-finger drag-select for free).

A second, older video-only library (`Pages/Video/VideoLibraryPage.swift`) duplicates this with uncached `AsyncImage` and should be consolidated or deprecated.

### Gap analysis

| Capability | Current | Effort to target |
|---|---|---|
| Dense adaptive grid, minimal gutters | ✅ Mostly there (fixed 3-col; sizing from static `Screen.bounds`, not rotation-safe) | S |
| **Pagination / large libraries** | ❌ **Hard 100-item cap** — `MediaActions.swift:71` fetches one page; server already supports `page`/`limit`; libraries >100 silently truncated | S–M |
| Pinch-to-zoom column levels | ❌ None on grid | M |
| Tap-to-expand into full-screen **pager** | ◐ Excellent single-item zoom transition; **no swipe-between-items** | M–L |
| Date sections + sticky headers | ❌ Flat list | M |
| Downsampled thumbs + prefetching | ◐ Cache exists; **no downsampling** (full decode), no `UICollectionViewDataSourcePrefetching`, no request coalescing; grid uses 400px `-md` variant when a 150px `-thumb` variant exists unused | S–M |
| Multi-select / drag-select | ❌ None (delete not even exposed in new grid) | M |
| Scroll position preservation | ◐ Full `reloadData()` on every change (`MediaLibraryGrid.swift:266-275`) — flashes, loses semantics | S (diffable data source) |
| Pull-to-refresh on media tab | ❌ Missing (Programs tab has it) | S |

Other notable issues: cell-hide/restore between grid and detail overlay handshakes via `NotificationCenter` (fragile, breaks with a pager); the "Create New → Media" upload entry is a TODO stub (`MainLibrary.swift:253-255`); `orderedMedia`/`filteredMedia` re-sort the full array per render.

### Recommended redesign — three phases

- **Phase 1 — Foundation (no visible redesign):** pagination with append semantics + `willDisplay` next-page trigger; `UICollectionViewDiffableDataSource` replacing `reloadData()`; `ImageCache.fetch(url:targetSize:)` with `CGImageSourceCreateThumbnailAtIndex` downsampling + in-flight coalescing; adopt prefetching; switch grid cells to `-thumb` variant (~7× fewer decoded pixels); add refresh control; migrate or deprecate `VideoLibraryPage`.
- **Phase 2 — Photos-grade grid:** compositional layout with date sections + pinned sticky headers; pinch zoom levels (1/3/5 columns) anchored on the pinch centroid, persisted in `FilterState`; multi-select mode with two-finger drag-select + bulk delete/tag bar.
- **Phase 3 — Pager & transition polish:** extract the detail view's media area into `MediaPageItemView` inside a horizontal `MediaPagerView` with ±1 neighbor prefetch; generalize the zoom transition into a `MediaZoomTransitionDriver` whose dismiss lands on the *currently paged* item's cell (scrolling the grid if needed, exactly like Photos) — replacing the NotificationCenter coupling; optional iOS 18 `.navigationTransition(.zoom)` feature flag with the custom driver as the iOS 17 fallback.

Proposed components: `MediaGridView` (SwiftUI shell) → `MediaGridController` (compositional layout + diffable source, zoom level, prefetch, selection) → `MediaCell`, `MediaSectionHeader`, `MediaPagerView`, `MediaPageItemView`, `MediaZoomTransitionDriver`.

## Prioritized Roadmap

### Tier 0 — Quick wins (hours each, do immediately)
1. Remove `NSAllowsArbitraryLoads` from Info.plist.
2. Move session cookie to Keychain (~30 lines, AuthManager is sole owner).
3. Delete cookie/APNs-token log lines (5 edits).
4. Replace the 16 `try!` sites with a safe markdown helper.
5. Fix CLAUDE.md ports/URLs/Actions table; remove the phantom `API_REFERENCE.md` mandate.
6. `Date+Formatting.swift` with cached static formatters (mechanical migration of 47 sites).
7. Default `defaultLocalIP` away from the hardcoded home LAN IP.

### Tier 1 — Compiler-enforced architecture (1–2 weeks)
1. `@MainActor` on the entire State layer (AppState, EntityStore, RelationshipIndex, LoadingStateManager).
2. Fix `StatePersistence` (latest-snapshot debounce, off-main encode).
3. DI for Actions (`init(api:state:)` + `APIClientProtocol`) and the first unit-test suite: EntityStore, persistence round-trip, model-decoding fixtures.
4. Rehome the 15 stray `APIClient` calls into Actions; route Bible services and `InviteQRCodeView` through APIClient.
5. `ToastManager` error channel; Actions become the catch-and-surface choke point.

### Tier 2 — Navigation overhaul (the transitions fix; 2–4 weeks, incremental)
Execute the 5-step migration in §2: Motion tokens → SlideStack → Route enum → NavigationCoordinator → escape-hatch sweep. Each step ships independently and each directly attacks a documented failure class.

### Tier 3 — Media library phases 1–3 (§3) + AuthManager modernization
Phase 1 of the media plan (pagination, diffable, downsampling) is safe to run in parallel with Tier 1 since it's UIKit-isolated. AuthManager `@Observable` conversion lands best after Tier 1 #4.

### Tier 4 — Design system enforcement
SwiftLint with custom rules (no print/NSLog, no force ops outside previews, no `Color(hex:)` outside Colors.swift, no raw font sizes), `Typography.swift` with Dynamic Type, accessibility-label checklist, fixtures behind `#if DEBUG`.

## Proposed Claude Code Skills

The transitions docs prove inspection-based enforcement has failed (their own rules contain exceptions, and the codebase violates its own MODAL_GUIDE). Skills move enforcement into the workflow. Recommended build order:

| # | Skill | Trigger | What it enforces / automates |
|---|---|---|---|
| 1 | **`/animation-debug`** | "X pops in / doesn't slide / flickers / jitters during transition" | Decision tree mapping symptom → the 10 known failure classes → known fix, citing the doc section and a reference implementation file. Highest immediate ROI: ends rediscovering diagnoses. |
| 2 | **`/transition-review`** | Any diff touching `withAnimation`, `.animation(`, `.transition(`, `offset(`, `DragGesture`, `asyncAfter` | Combined checklist from both scar-tissue docs: mixed curves, duplicate `.animation` modifiers, `@State` drag tracking, missing `.compositingGroup()`, `@State` mutation in `.task`, conditional always-visible layers, LazyVStack in animated containers, hardcoded durations (must use `Motion` tokens once they exist). |
| 3 | **`/push-page`** | "Add a detail/edit screen", any new sub-screen inside a modal | Scaffolds with canonical `SlideStack` (post-migration) or emits the exact two-step + `.animation(value:)` + `.buttonStyle(.plain)` recipe (pre-migration); forbids new hand-rolled HStack/offset navigation; checks cache pre-load in `init` and VStack-not-LazyVStack. |
| 4 | **`/present-overlay`** | Any new modal, menu, or sheet | `presentModal`/`presentMenu` over `fullScreenCover`/`sheet` (with the hardware-access exception list); registers the Route/OverlayID; wires environment dismissal; rejects dismiss-then-present without completion handling. |
| 5 | **`/nav-route`** | Deep link, push-notification destination, cross-tab jump | Typed `Route` through the coordinator instead of `currentTab`/Int pokes; updates `handleDeepLink` exhaustively. (Build after Tier 2 step 3.) |

Skills 1–2 are buildable **today** from the existing docs; 3–5 get sharper as the Tier 2 migration lands. A sixth candidate — `/ios-error-surface` (every new catch must route to ToastManager or justify swallowing) — becomes viable after Tier 1 #5.
