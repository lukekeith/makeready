## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Removed @apply from admin.css base layer**
- **Found during:** Task 2 (build verification)
- **Issue:** `@apply border-border`, `@apply bg-background`, `@apply text-foreground` caused Tailwind v4 `onInvalidCandidate` build error — CSS variable-based utility classes are not available as @apply targets without explicit @theme mapping
- **Fix:** Replaced `@apply` rules with plain CSS (`border-color: hsl(var(--border))`, `background-color: hsl(var(--background))`, `color: hsl(var(--foreground))`)
- **Files modified:** resources/css/admin.css
- **Commit:** ca9fbb6

**2. [Rule 2 - Missing Config] Added @theme inline block to admin.css**
- **Found during:** Task 1 analysis
- **Issue:** Tailwind v4 does not automatically map CSS custom properties in `:root` to utility classes — `bg-primary`, `text-foreground` etc. in shadcn components would not generate without explicit `@theme inline` declarations
- **Fix:** Added `@theme inline` block mapping all color CSS variables via `hsl(var(...))` pattern
- **Files modified:** resources/css/admin.css
- **Commit:** ca9fbb6

**3. [Rule 3 - Blocking] Manual components.json creation (skipped interactive CLI)**
- **Found during:** Task 1
- **Issue:** `npx shadcn-vue@latest init` is interactive and cannot be automated — the `--yes` flag skipped init without creating components.json
- **Fix:** Created components.json manually with correct schema (new-york style, zinc, cssVariables, correct aliases). Also removed invalid `framework` key which caused a schema validation error on first attempt.
- **Files modified:** components.json
- **Commit:** b323a3f
