## Summary

Phase 12.1 replaces 8 BEM admin Vue components with shadcn-vue equivalents, deletes 16 SCSS files, and installs Tailwind CSS v4 scoped exclusively to the admin island. The project already uses reka-ui (the same underlying primitive library shadcn-vue uses), so shadcn-vue installation does not introduce duplicate primitive layers — it wraps the same reka-ui components in pre-styled, themed compositions.

The key technical complexity is ensuring Tailwind v4 does not contaminate existing Blade/SCSS pages. Tailwind v4's `source(none)` + explicit `@source` directives solve this at the CSS level: `admin.css` only generates utilities for files in the admin island paths. The admin Blade layout loads `admin.css` as a second Vite entry alongside `app.scss`.

The shadcn-vue Sidebar (sidebar-07 pattern) is the most structurally significant change: it requires wrapping `admin-island.vue` in `SidebarProvider`, replacing the existing hand-rolled expand/collapse logic entirely. All other components are close 1-to-1 replacements.

**Primary recommendation:** Install Tailwind v4 with `@tailwindcss/vite` first, create `admin.css` with `source(none)` scoping, configure `components.json` with the correct aliases, then run `npx shadcn-vue@latest add` to install all required components in one pass before migrating sections one at a time.

### Pattern 1: Tailwind CSS v4 Scoped Entry Point

**What:** A dedicated `admin.css` file that imports Tailwind with `source(none)` and explicitly registers only admin-island source paths. This prevents any Tailwind utilities from being generated for Blade/SCSS pages.

**When to use:** Any project with mixed CSS approaches (SCSS/BEM + Tailwind) sharing the same Vite build.

```css
/* resources/css/admin.css */
/* Source: https://tailwindcss.com/docs/detecting-classes-in-source-files */
@import "tailwindcss" source(none);

/* Only scan admin-specific files */
@source "../js/islands/admin-island/**/*.vue";
@source "../js/components/admin/**/*.vue";
@source "../js/components/ui/**/*.vue";
```

**vite.config.js addition:**
```javascript
// Source: official Laravel Vite plugin docs
laravel({
    input: [
        'resources/css/app.scss',
        'resources/css/admin.css',   // NEW
        'resources/js/app.js'
    ],
    refresh: true,
}),
tailwindcss(),   // ADD the @tailwindcss/vite plugin
```

**Full updated vite.config.js:**
```javascript
import path from 'path'
import { defineConfig } from 'vite'
import laravel from 'laravel-vite-plugin'
import vue from '@vitejs/plugin-vue'

                'resources/css/app.scss',

```

**admin.blade.php addition:**

### Pattern 2: shadcn-vue Initialization for Non-Standard Laravel Structure

**What:** The `components.json` must be configured to output components to `resources/js/components/ui/` rather than the default `src/components/ui/`. The `@` alias is already configured in `tsconfig.json` as `resources/js/*`.

**components.json configuration:**
```json
{
  "$schema": "https://shadcn-vue.com/schema.json",
  "style": "new-york",
  "typescript": true,
  "tailwind": {
    "config": "",
    "css": "resources/css/admin.css",
    "baseColor": "zinc",
    "cssVariables": true,
    "prefix": ""
  },
  "aliases": {
    "components": "@/components",
    "composables": "@/composables",
    "utils": "@/lib/utils",
    "ui": "@/components/ui",
    "lib": "@/lib"
  }
}
```

**Key:** `aliases.ui = "@/components/ui"` tells the CLI to write components to `resources/js/components/ui/`. The `@` alias resolves to `resources/js/` per `tsconfig.json`.

### Pattern 3: Forced Dark Mode via `.dark` Class

**What:** shadcn-vue's CSS variables respond to the `.dark` class on any ancestor element. To force dark mode for the admin panel without affecting member-facing pages, apply `.dark` to the admin island wrapper rather than `<html>`.

**When to use:** Mixed light/dark apps where admin needs permanent dark mode.

```vue
<!-- admin-island.vue — add class="dark" to root element -->
<template>
  <SidebarProvider>
    <div class="dark AdminLayout">
      <AppSidebar ... />
      <SidebarInset>
        <RouterView />
      </SidebarInset>
    </div>
  </SidebarProvider>
</template>
```

This scopes all zinc dark CSS variables to admin without touching Blade/SSR pages.

**Zinc dark theme key variables (OKLCH format, Tailwind v4):**
- `--background: oklch(0.141 0.005 285.823)` — nearly same as current `#1a1d2e`
- `--foreground: oklch(0.985 0 0)` — white text
- `--card: oklch(0.21 0.006 285.885)` — card backgrounds
- `--border: oklch(1 0 0 / 10%)` — subtle borders

      <TableRow
        v-for="virtualRow in virtualizer.virtualItems"

### Pattern 8: shadcn Form with vee-validate + Zod

**What:** shadcn-vue's `<Form>` component is a direct abstraction over vee-validate. It requires `vee-validate`, `@vee-validate/zod`, and `zod` (already installed). Each field uses `FormField`, `FormItem`, `FormLabel`, `FormControl`, `FormMessage`.

```vue
<script setup lang="ts">
import { useForm } from 'vee-validate'
import { toTypedSchema } from '@vee-validate/zod'
import * as z from 'zod'
import { Form, FormControl, FormField, FormItem, FormLabel, FormMessage } from '@/components/ui/form'
import { Input } from '@/components/ui/input'
import { Button } from '@/components/ui/button'

const formSchema = toTypedSchema(z.object({
  name: z.string().min(1, 'Name is required'),
}))

const form = useForm({ validationSchema: formSchema })
const onSubmit = form.handleSubmit((values) => emit('save', values))
</script>

<template>
  <form @submit="onSubmit">
    <FormField v-slot="{ componentField }" name="name">
      <FormItem>
        <FormLabel>Name</FormLabel>
        <FormControl>
          <Input v-bind="componentField" placeholder="Enter name" />
        </FormControl>
        <FormMessage />
      </FormItem>
    </FormField>
    <Button type="submit" :disabled="saving">Save</Button>
  </form>
</template>
```

**Migration note for AdminForm:** Current `admin-form.vue` receives a generic `fields: FormField[]` array and renders inputs dynamically. The shadcn Form migration can preserve this pattern by iterating `fields` and rendering the appropriate shadcn input component per field type.

### Anti-Patterns to Avoid

- **Mixing shadcn `<Form>` and bare `<input>` elements:** Form accessibility relies on shadcn's `FormControl` > `FormItem` structure. Bare inputs break the ARIA labeling chain.
- **Adding `.dark` to `<html>` or `<body>`:** This would force dark mode on Blade/member pages. Scope `.dark` to the admin island container only.
- **Using `@apply` in Vue `<style scoped>` blocks:** Tailwind v4 has a known issue where `@apply` in Vue scoped styles does not work in production builds. Use Tailwind utility classes directly in templates instead.
- **Forgetting `SidebarProvider`:** The `Sidebar` component throws if not wrapped in `SidebarProvider`. It must be the root wrapper in `admin-island.vue`.
- **Placing shadcn components outside `resources/js/components/ui/`:** The CLI writes to the path in `aliases.ui`. Manually placing components elsewhere breaks the `npx shadcn-vue add` update path.
- **Running `npx shadcn-vue@latest add` before `tailwindcss/vite` is configured:** The CLI writes CSS variable definitions into the CSS file specified in `components.json`. That file must exist and import Tailwind before the CLI runs.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Slide-over drawer with backdrop + animation | Custom Teleport + CSS transition | `<Sheet side="right">` | Sheet handles focus trap, escape key, backdrop, animation, accessibility |
| Confirm dialog with focus trap | `<DialogRoot>` from reka-ui directly | `<AlertDialog>` | AlertDialog is semantically correct for destructive confirmations; includes ARIA role="alertdialog" |
| Sidebar collapse logic | `ref(isExpanded)` + manual CSS classes | `SidebarProvider` + `useSidebar()` | Provider handles persistent state, keyboard shortcut (Cmd+B), mobile responsive, cookie persistence |
| Form field validation state | Per-field `ref(error)` + manual ARIA | `<FormField>` + `<FormMessage>` | shadcn Form propagates vee-validate state to ARIA attributes automatically |
| Searchable dropdown/combobox | Popover + manual input filter logic | `<Combobox>` (Popover + Command) | Command component includes built-in search, keyboard navigation, empty states |
| Loading shimmer animation | Custom CSS keyframe | `<Skeleton>` | One-liner; animates correctly in dark mode with `--muted` variable |

**Key insight:** Every existing BEM component in this admin island is wrapping reka-ui primitives manually (see `admin-confirm-dialog.vue` using `DialogRoot, DialogPortal, DialogOverlay` directly). shadcn-vue wraps those same primitives with pre-built styles and accessibility. The migration is removing the manual wrapper code, not changing the underlying behavior.

## Common Pitfalls

### Pitfall 1: `@apply` Breaks in Vue Scoped Styles with Tailwind v4
**What goes wrong:** Any `@apply` directive inside a Vue component's `<style scoped>` block fails silently in development and breaks visually in production builds.
**Why it happens:** Tailwind v4's Vite plugin processes `@apply` differently than PostCSS; Vue's scoped CSS transform conflicts with the Tailwind Vite transform.
**How to avoid:** Never use `@apply` in `.vue` files. Apply all Tailwind utility classes directly in the template. If shared styles are needed, define them in `admin.css` using `@layer components`.
**Warning signs:** Styles appear correct in dev but disappear after `vite build`.

### Pitfall 2: Tailwind Utility Classes Leaking into Blade Pages
**What goes wrong:** Tailwind generates utilities that affect elements on Blade pages with matching class names (e.g., `text-sm`, `flex`, `p-4` if they happen to appear in SCSS variables or partials).
**Why it happens:** Without `source(none)`, Tailwind v4's auto-detection scans the project broadly.
**How to avoid:** `admin.css` MUST use `@import "tailwindcss" source(none)` with explicit `@source` paths covering only admin Vue files.
**Warning signs:** Member-facing Blade pages render differently after adding `admin.css` to the Vite build.

### Pitfall 3: Dark Mode Applies Globally
**What goes wrong:** Member pages get dark background if `.dark` class is added to `<html>` or `<body>`.
**Why it happens:** CSS variables are scoped to `.dark {}` selector which matches any ancestor, including `<html>`.
**How to avoid:** Apply `class="dark"` ONLY to the `<SidebarProvider>` or root div inside `admin-island.vue`. Member Blade pages load `app.scss` without shadcn variables — they are unaffected as long as `.dark` is not on `<html>`.
**Warning signs:** `/join`, `/groups`, or other member pages show dark backgrounds.

### Pitfall 4: `SidebarProvider` Missing from `admin-island.vue`
**What goes wrong:** Vue runtime error: `useSidebar() must be used within a SidebarProvider`.
**Why it happens:** Every shadcn Sidebar sub-component calls `useSidebar()` which reads from a `provide`/`inject` context set by `SidebarProvider`.
**How to avoid:** `SidebarProvider` must be the outermost wrapper in `admin-island.vue` before `<AdminSidebar>` is rendered.
**Warning signs:** Console error on admin page load; entire admin island fails to mount.

### Pitfall 5: TanStack Virtual Row Spacers Break shadcn TableRow
**What goes wrong:** The spacer-row pattern (empty `<tr>` with explicit pixel height for virtual scroll positioning) conflicts with shadcn `<TableRow>` styling which adds hover/border styles.
**Why it happens:** shadcn `<TableRow>` applies `border-b` and hover background to all rows including spacers.
**How to avoid:** Render spacer rows as bare `<tr>` elements (not `<TableRow>`) or add `class="border-0 hover:bg-transparent pointer-events-none"` to spacer `<TableRow>` components.
**Warning signs:** Phantom hover highlight on empty space above/below virtual rows.

### Pitfall 6: `components.json` Path Aliases Mismatch
**What goes wrong:** `npx shadcn-vue@latest add` writes component files to wrong directory or imports fail with "Cannot find module".
**Why it happens:** `components.json` `aliases.ui` must match a path alias configured in `tsconfig.json`. This project uses `@` = `resources/js/`, so `aliases.ui = "@/components/ui"` is correct.
**How to avoid:** Verify `tsconfig.json` paths before running `shadcn-vue init`. The `@` alias maps to `resources/js/` — confirm `components.json` uses this consistently.
**Warning signs:** TypeScript errors on shadcn component imports; files created in unexpected locations.

### Pitfall 7: AdminForm Migration — Dynamic Fields Array
**What goes wrong:** Current `admin-form.vue` accepts a generic `fields: FormField[]` array and renders inputs dynamically. shadcn's `<FormField>` uses a `name` string prop tied to the vee-validate schema, not a dynamic array index.
**Why it happens:** vee-validate's `useField` requires a static field name registered against the schema at component setup time.
**How to avoid:** Two options — (a) keep the dynamic `fields` prop but use shadcn Input/Select/Textarea components without the Form wrapper for purely visual consistency, OR (b) replace generic AdminForm with per-section inline forms that use static field names. Option (a) is safer for this phase scope.
**Warning signs:** vee-validate emits "Field is not defined in the schema" warnings for dynamic field names.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `tailwind.config.js` + `content: []` array | `@import "tailwindcss" source(none)` + `@source` directives in CSS | Tailwind v4 (Jan 2025) | No JS config file needed; scoping is CSS-native |
| PostCSS plugin for Tailwind | `@tailwindcss/vite` Vite plugin | Tailwind v4 (Jan 2025) | Faster builds, simpler config |
| Radix-Vue as shadcn-vue primitive | Reka-UI as shadcn-vue primitive | shadcn-vue v1 (Feb 2025) | This project already uses reka-ui v2.9.2 — no conflict |
| HSL color variables in shadcn | OKLCH color variables | shadcn-vue v1 + Tailwind v4 | CSS variables have same semantic names; template code unchanged |
| `shadcn-vue@radix` for Radix-based install | `shadcn-vue@latest` for Reka-UI-based install | Feb 2025 | Must use `@latest` to get reka-ui components; `@radix` installs old version |

**Deprecated/outdated:**
- `tailwind.config.js` with `content: []` array: Replaced by CSS `@source` directives in Tailwind v4. Do not create `tailwind.config.js` for this phase.
- `@tailwindcss/postcss`: Replaced by `@tailwindcss/vite` for Vite-based projects. Do not add to PostCSS config.
- `shadcn-vue@radix`: This pin installs the Radix-Vue-based version; use `shadcn-vue@latest` to get Reka-UI components which align with this project's existing `reka-ui` dependency.

## Open Questions

1. **Does `admin-form.vue`'s dynamic `fields: FormField[]` pattern need to be preserved exactly?**
   - What we know: shadcn Form wraps vee-validate which requires static field names per schema
   - What's unclear: Whether section views (groups-section, programs-section) rely on the dynamic fields API pattern or could be refactored to inline forms
   - Recommendation: Use shadcn visual components (Input, Select, Button) without the Form/vee-validate layer for AdminForm to preserve the dynamic fields interface. Add vee-validate-backed forms only for truly new form surfaces.

2. **Does `admin.css` get injected before or after `app.scss` in the admin layout?**
   - What we know: Blade layout currently loads `app.scss` + `app.js` together
   - What's unclear: Whether shadcn CSS variable definitions in `admin.css` need to load before any admin component renders, or whether Tailwind's `@import "tailwindcss"` and app.scss interact at the `:root` level
   - Recommendation: Load `admin.css` after `app.scss` in `admin.blade.php`. Both define variables at `:root` scope — shadcn variables will override any conflicting app.scss root variables, but app.scss defines none that conflict with `--background`, `--foreground` etc.

3. **ApexCharts dark theme integration**
   - What we know: Charts currently use hardcoded inline styles (`style="color:#7c3aed"`) and BEM container classes
   - What's unclear: Whether ApexCharts needs theme configuration to match shadcn zinc dark colors
   - Recommendation: Pass `theme: { mode: 'dark' }` in ApexCharts `options` objects in analytics.ui store. This is a store-level change (adding one option), not a template change.
