## Summary

Phase 3.1 is a gap-closure sprint that fixes the visual mismatch discovered during Phase 4 testing. The root cause is two-fold: (1) page-level SCSS files from `archive/react-spa:src/pages/*/` were never migrated to the Laravel pipeline — they do not exist at all in `resources/css/` — and (2) some Blade page templates use different root CSS class names than the React pages, meaning even if the SCSS existed it would not apply. A secondary issue is that several Blade page templates restructure the HTML composition differently from React (e.g., adding section headings or div wrappers that React did not have), which breaks BEM selectors.

Component-level HTML structure is largely accurate. A survey of the Button, GroupCard, and GroupListCard Blade components against their React TSX shows the BEM class names and DOM nesting are correct. The main gap at the component level is the Button's inner content structure: React always wraps content in a `Button__content` span, and the Jump variant wraps label/description in a `Button__details` span. The current Blade Button only wraps content in `Button__content` for the loading state, and does not render `Button__details` at all. The `button.scss` expects these elements.

The work decomposes naturally into three tracks: (A) migrate 14 missing page-level SCSS files and wire them into `app.scss`; (B) rewrite Blade page templates whose root class names diverge from React; (C) fix component HTML structure where the Blade output does not match React's rendered DOM. Track A is mechanical and fast. Track B and C require per-file comparison against the archive branch.

**Primary recommendation:** Work page-by-page using the archive branch as the authoritative reference. For each page: (1) `git show origin/archive/react-spa:src/pages/<name>/<name>.page.tsx` to read the React JSX, (2) compare to the current Blade template, (3) rewrite the Blade to match class names and composition, (4) `git show origin/archive/react-spa:src/pages/<name>/<name>.scss` to copy the SCSS into `resources/css/pages/`, (5) add `@use 'pages/<name>'` to `app.scss`.

### Pattern 3: HTML Composition Matching

**What:** When a React page renders components directly (e.g., `<StudyCard mode="Header">` inside `StudyHome__scroll-container`), the Blade must match that exact nesting. Avoid adding intermediate wrappers or semantic elements (`<section>`, `<h2>`) that React did not render.

**Example gap found — `study-home.blade.php`:**

React structure:
```
.StudyHome
  .StudyHome__viewport
    .StudyHome__scroll-container
      .StudyHome__header           ← div, not <section>
        <StudyCard mode="Header">
        <StudyScheduleCard>
        .StudyHome__lessons
          <StudyCard mode="LessonList"> × n
      .StudyHome__bottom-spacer
    .StudyHome__navigation
```

Current Blade structure has:
```
.StudyHomePage                     ← wrong root class
  <navigation>
  <page-title>
  .StudyHomePage__container        ← wrong nesting
    <section class="StudyHomePage__progress">   ← section instead of div
    <section class="StudyHomePage__lesson-list"> ← added section wrapper
      <h2>Lessons</h2>             ← React never renders this h2
```

**Fix:** Rewrite the Blade to match the React structure exactly.

### Pattern 4: Button Inner Structure

**What:** The React Button component always wraps its content in `Button__content`. Jump variants additionally wrap label+description in `Button__details`. The Blade currently only uses `Button__content` in the loading branch. The button SCSS requires these elements.

**React Button render output:**
```html
<!-- Standard button with label -->
<button class="Button Button--primary Button--size-default Button--mode-action">
  <span class="Button__content">
    <span class="Button__label">Save</span>
  </span>
</button>

<!-- Jump variant -->
<button class="Button Button--jump">
  <span class="Button__content">
    <span class="Button__details">
      <span class="Button__label">Member Login</span>
      <span class="Button__description">Sign in with your phone number</span>
    </span>
    <span class="Button__icon Button__icon--right">...</span>
  </span>
</button>
```

**Current Blade (incorrect):**
```html
<button class="Button Button--primary Button--size-default Button--mode-action">
  Save                         ← no Button__content wrapper
</button>
```

**Fix:** Always wrap content in `Button__content`, implement `Button__details` for Jump variants, implement `Button__icon` spans for left/right icon props.

### Anti-Patterns to Avoid

- **Adding structural wrappers React doesn't have:** Do not add `<section>`, `<h2>`, or extra `<div>` elements around content groups unless React renders them. Check the archive TSX first.
- **Changing root class names:** Every page's outermost element must use exactly the class name from the React TSX, not a Blade-invented name.
- **Skipping SCSS migration:** Never fix the HTML without also migrating the corresponding SCSS. Without the SCSS, the page has no styles at all.
- **Partial SCSS migration:** Do not copy only part of a page's SCSS. Copy the entire file verbatim from archive.

## Common Pitfalls

### Pitfall 1: Incorrect Root Class Name

**What goes wrong:** The Blade page renders with a class like `StudyHomePage` but the SCSS file (from archive) defines `.StudyHome`. The SCSS rules never match the HTML so the page has no layout.

**Why it happens:** Blade files were created with Blade-conventional naming (appending `Page`) rather than preserving the exact React class name.

**How to avoid:** Always check the React TSX first. The root JSX element's `className` is the class the SCSS targets.

**Warning signs:** Page renders with no layout structure despite SCSS being imported.

### Pitfall 2: Extra Wrapper Elements Breaking BEM

**What goes wrong:** Adding `<section class="GroupHome__studies">` with `<h2 class="GroupHome__section-title">` inside a container that React renders as a plain `<div>`. The inner child selectors in SCSS do not expect the extra heading element, causing layout gaps.

**Why it happens:** Blade rewrites were done semantically (adding heading tags for accessibility) rather than by reading the React TSX spec.

**How to avoid:** Read the React TSX for each page section before writing the Blade equivalent.

**Warning signs:** Vertical spacing or alignment is off even when root class and SCSS are correct.

### Pitfall 3: Missing Page SCSS (No Styles at All)

**What goes wrong:** Page renders but has no visual styling — wrong background color, no layout, elements stack vertically without padding.

**Why it happens:** `resources/css/pages/` does not exist yet. Page-level SCSS was never migrated. Only component SCSS exists.

**How to avoid:** Create `resources/css/pages/` directory, copy the SCSS file from archive for each page, and add `@use 'pages/<name>'` to `app.scss`.

**Warning signs:** Page has correct HTML class attributes but default browser styling instead of dark theme.

### Pitfall 4: Button Content Not Wrapped

**What goes wrong:** Buttons render with no label or appear broken because the SCSS targets `.Button__content` which the Blade does not emit.

**Why it happens:** The current Blade Button only uses `Button__content` in the loading path; the normal label path renders text directly inside `<button>`.

**How to avoid:** Always wrap all button content in `<span class="Button__content">`. Use `Button__details` for Jump variant label/description.

**Warning signs:** Button labels don't align, Jump buttons don't show description text, icon buttons are unsized.

### Pitfall 5: Sass @use Load Path Required for Pages

**What goes wrong:** Adding `@use 'pages/home'` to `app.scss` fails with "file not found" because Sass resolves relative to the file's own directory.

**Why it happens:** The existing `@use 'components/...'` entries work because `resources/css/components/` exists. Once `resources/css/pages/` is created, the path will resolve the same way.

**How to avoid:** Create `resources/css/pages/` directory before adding `@use` entries to `app.scss`. The `vite.config.js` already has the correct `loadPaths` configuration — no changes needed there.

**Warning signs:** Vite build error: "Can't find stylesheet to import."

### Recommended Test Approach

Visual CSS correctness cannot be asserted via PHPUnit (no browser). The practical testing strategy is:

1. **Class name correctness (automated):** Add assertions to existing feature tests that the correct BEM root class appears in the rendered HTML. E.g., `$response->assertSee('class="MemberHomePage"', false)`.

2. **Vite build passes (automated):** `npm run build` must complete without Sass compilation errors. This catches missing `@use` paths and SCSS syntax errors.

3. **Visual comparison (manual gate):** Run dev server, open each page in browser, compare side-by-side with the React app running from the archive branch or screenshots. This is the ultimate fidelity gate.
